# 发布准入指标表

本文件定义了各风险等级下，所有测评指标的准入基线。测评完成后，对照此表做发布决策。

>

---

## 核心概念澄清

**对比基线（Comparison Baseline）≠ 准入基线（Admission Baseline）**

- **对比基线**：without_skill 通过率 52% → with_skill 通过率 87% → Δ = +35%
- **准入基线**：通过率必须 ≥ 90% 才允许发布（不管 Δ 是多少）

Δ 很大但绝对值不够，仍然不能发布。反过来，Δ 很小但绝对值已经 95%，也可以发布。这两个概念必须分开看。

---

## 完整发布准入指标表

| 指标 | S级（关键） | A级（重要） | B级（一般） | C级（辅助） | 确定依据 |
|------|-----------|-----------|-----------|-----------|---------|
| **通过率** | ≥ 95% | ≥ 90% | ≥ 80% | ≥ 70% | 业务容错度：S级一次错误可能造成直接损失 |
| **触发率（精确）** | ≥ 95% | ≥ 90% | ≥ 85% | ≥ 80% | 触发失败=功能不可用，高风险容忍度更低。需 claude CLI 精确测量 |
| **触发率（AI估算）** ⭐ | TP ≥ 80%（参考） | TP ≥ 80%（参考） | TP ≥ 70%（参考） | 仅参考 |
| **增益 (Δ)** | > 0，不允许负向 | > 0 | ≥ -5%（允许小幅负向） | 不硬性要求 | S/A级任何负向都不可接受 |
| **指令遵循率（IFR）** | = 100% | ≥ 95% | ≥ 90% | ≥ 80% | S级硬性规则是合规底线，不能有遗漏 |
| **一致性得分** | ≥ 95% | ≥ 90% | ≥ 80% | 不硬性要求 | 输出不一致=用户体验不可预测 |
| **稳定性 (Stddev)** | < 0.05 | < 0.10 | < 0.20 | < 0.30 | 高风险 Skill 输出必须高度稳定 |
| **测评覆盖率** | ≥ 95% | ≥ 85% | ≥ 70% | ≥ 50% | 高风险 Skill 不允许有未测到的规则 |
| **灾难场景** | 全部通过（红线） | 全部通过（红线） | 不强制 | 不要求 | 灾难场景是红线，一票否决 |
| **安全测试** | 全部通过 | 全部通过 | 基础项通过 | 不要求 | 安全是底线 |
| **幻觉检测** | 0 幻觉 | ≤ 1 次 | ≤ 2 次 | 不硬性要求 | S级不允许任何编造信息 |
| **P95 响应时间** ⭐ | < 15s | < 15s | < 30s | < 30s |
| **额外 Token 消耗** 🆕 | ≤ 2000 tokens/用例（警告） | ≤ 2000 tokens/用例（警告） | ≤ 3000 tokens/用例（警告） | 不硬性要求 |
| **复杂度得分** 🆕 | ≤ 20（警告）/ ≤ 30（红线） | ≤ 20（警告）/ ≤ 30（红线） | ≤ 30（警告） | 不硬性要求 |
| **HiL 确认节点** 🆕 | 不可逆操作必须有（⚠️警告） | 不可逆操作必须有（⚠️警告） | 建议有 | 不要求 |

> ⭐ = 性能指标 · 🆕 = 效率/成本指标

## 触发率指标的两套测量方案

触发率测评提供两套方案，根据可用环境选择：

### 方案 A：AI 模拟估算（默认）

**工作原理**：
1. 从 `description` 字段提取触发语义
2. AI 生成 10 条测试 prompt（5 TP + 3 TN + 2 边界）
3. AI 判断每条 prompt 的触发概率，输出 `trigger_eval.json`
4. 报告第十一章可视化展示，标注置信度（high/medium/low）

**使用场景**：日常测评、快速预判 description 质量、无 claude CLI 环境时

**局限性**：估算值，置信度有限，S/A 级正式发布前必须使用方案 B 精确测量

**准入处理（P4 门槛规则）**：

| 情况 | S/A 级影响 | B/C 级影响 |
|------|----------|----------|
| TP 估算 ≥ 80%，置信度 high/medium | 无影响，标注参考值 | 无影响 |
| TP 估算 ≥ 80%，置信度 **low** | **强制降为 CONDITIONAL PASS** | ⚠️ 警告 |
| TP 估算 70%-80% | ⚠️ 警告，建议优化 description | ⚠️ 警告 |
| TP 估算 < 70% | **强制降为 CONDITIONAL PASS** | ⚠️ 警告 |
| TN 有误触发预测 | **强制降为 CONDITIONAL PASS** + 标红 | ⚠️ 警告 |

> **为什么 low 置信度也触发降级**：置信度 low 意味着 description 本身表达不清晰，AI 都判断不了该不该触发，真实用户更难判断。这类 Skill 的触发行为不可预测，不应发布。

### 方案 B：精确测量（需 claude CLI）

通过 skill-creator 的 `run_eval.py` 真实运行测试：
- 每条 query 运行 3 次，取均值
- 10 个 worker 并行执行
- 产出精确触发率，可用于正式发布决策

**准入处理**：按完整发布准入指标表中「触发率（精确）」行执行，硬性阻止发布。

## 断言强度分级与通过率计算（P1）

断言按 `precision` 字段分级，准入判断只看 `exact_match` 通过率：

| precision | 含义 | 计入准入判断 |
|-----------|------|------------|
| `exact_match` | 有具体可验证的字段值/计数/格式 | **是（核心指标）** |
| `semantic` | 需要语义理解 | 是（参考） |
| `existence` | 只验证存在/不存在，无 Skill 也大概率通过 | **否** |

**报告展示三个通过率**：
- `精确通过率`（exact_match）= 准入判断依据
- `语义通过率`（semantic）= 辅助参考
- `综合通过率`（all）= 兼容旧版，标注构成

## 发布决策矩阵

根据指标达成情况，给出以下三种决策：

### PASS（可以发布）
- 所有硬性指标达标（以 exact_match 通过率为准）
- 无 Δ < 0 的测试用例
- S/A级：灾难场景全部通过
- 触发率 AI 估算 ≥ 80% 且置信度 ≥ medium（S/A 级）
- 人工评审无重大问题

### CONDITIONAL PASS（有条件发布）
- exact_match 通过率在目标阈值 ±5% 范围内（如目标 90%，实际 86-89%）
- 有 1-2 个 Δ < 0 的用例，但根因已明确且影响有限
- 触发率 AI 估算满足降级条件（置信度 low / TP < 70% / TN 误触发）
- quick 模式两次运行差距 > 15%（结果不稳定）
- 达成约定：在上线后 X 天内迭代修复，重新测评
- 需要产品/研发负责人明确知情并签字

### FAIL（不允许发布）
- 核心指标（exact_match 通过率、IFR）明显未达标
- 存在未解决的 Δ < 0 用例
- S/A级：任何灾难场景失败
- 发现幻觉（S/A级超出阈值）
- 安全测试失败

## 准入基线的确定方法

这些数字不是拍脑袋的，基于以下三个因素：

### 1. 业务容错度（Business Tolerance）
- 这个 Skill 出一次错，代价是什么？
- 代价越大 → 基线越高

### 2. 用户预期（User Expectation）
- 用户把 Skill 当"工具"（直接提交）→ 基线必须高
- 用户把 Skill 当"参考"（看完再改）→ 基线可以稍低

### 3. 历史数据校准（Historical Calibration）
- 首版 Skill 基线可以相对宽松（先上线再迭代）
- 随数据积累，用真实失败率修正基线
- 线上投诉率 > 5% → 提高对应指标基线

**参考基线数据（em-reimbursement-v3，2026年3月，A级）**：

以下数据来自 sessions/ 目录中已完成的历史测评，供阈值校准参考：

| 测评日期 | 模式 | exact_match 通过率 | 增益 Δ | without_skill 耗时 | 备注 |
|---------|------|-------------------|-------|------------------|------|
| 2026-03-22 | quick（12用例） | ~75% | +40%+ | ~15-22 min | 首次测评，规则提炼阶段 |
| 2026-03-27 | quick（10用例） | ~80% | +35%+ | ~18 min | 含 AT/BD/DS/HP 四类用例 |
| 2026-03-29 | quick（部分） | 进行中 | — | — | 含触发率预评估 |

**阈值决策依据**：
- A级 exact_match 准入线 90% 源于：em-reimbursement-v3 历史通过率 75-80%，设置 90% 目标给出 10-15% 的改进空间，与"先上线后迭代"策略一致
- without_skill P95 耗时实测约 18-22 分钟（差旅报销类复杂 Skill），quick 模式总时长估算已基于此数据修正
- 这些数字会随测评次数增加持续校准，**不是永久固定值**

## 对比基线运行规范

### without_skill 基线

| 问题 | 规则 |
|------|------|
| 每次都要重新跑吗？ | 首次必须跑。模型版本没变可复用；版本变更必须重跑 |
| 跑几次？ | 与 with_skill 相同次数（建议 3 次），取均值 |
| 基线本身已经 > 85%，还有测评价值吗？ | 参考领域适配性预评估。Δ < 10% 时评估维护成本是否值得 |

### 改进 Skill 时的旧版本基线

| 问题 | 规则 |
|------|------|
| 什么时候要重跑旧版本基线？ | 模型版本或任何环境参数变化时，必须重跑 |
| 为什么不能跨版本对比？ | 不同环境的结果没有可比性，会导致错误结论 |

## 三条件对比范式（深度评测可选）

```
条件A：without_skill       —— 模型自然推理，无任何 Skill 指导
条件B：with curated_skill  —— 使用人工精心设计的 Skill（标准测评）
条件C：with self-generated_skill —— 让模型自己生成 Skill 再执行

解读：
  B > C：人工 Skill 有价值，人工编写优于模型自动生成
  B ≈ C：该任务 Skill 已接近模型上限，继续优化边际收益低
  B < C：极少见，说明 Skill 可能过度约束了模型
```
