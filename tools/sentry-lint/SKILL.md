# sentry-lint · Skill 静态质量检查

对目标 SKILL.md 做零执行的静态分析，输出结构化 lint 报告。约 30 秒完成。

**一句话价值**：发现「功能上能跑但设计有缺陷」的问题，比运行用例更快、成本更低。

---

## 输入

优先级：
1. 用户提供了具体路径 → 直接使用
2. 用户只说了名字 → 按顺序查找：`~/.claude/skills/<名字>/` → `~/.config/opencode/skills/<名字>/`
3. 用户说「检查这个 skill」→ 查找当前工作目录下的 SKILL.md

---

## 执行清单（逐项输出，不可跳过）

### L1：description 字段完整性

读取 SKILL.md frontmatter 中的 `description` 字段，逐项检查：

```
□ description 存在且非空
□ 包含「触发场景」描述（帮助模型判断何时激活）
□ 包含「不触发场景」描述（防止误触发）
□ 描述中有具体业务动词（如「报销」「创建单据」），而非泛化描述（如「帮助用户」）
□ 描述总长度 ≥ 50 字（过短则触发不准）
```

每项标注 ✅ / ⚠️ / ❌，并给出具体理由。

---

### L2：Human-in-the-Loop（HiL）节点检查

扫描 SKILL.md 全文，查找「不可逆操作」关键词：

```
不可逆操作关键词：提交、删除、创建、写入、发送、上传、支付、更新、修改、save、submit、delete、create、write、send
```

对每个发现的不可逆操作：

```
HiL-1：该操作前是否有明确的用户确认步骤？
  关键词：「询问用户」「等待确认」「用户批准」「是否继续」「please confirm」「wait for user」
  → 有：✅
  → 没有：❌ 标注「缺少 HiL 确认节点，操作：[具体操作描述]」

HiL-2：确认失败或用户拒绝时，是否有明确的中止逻辑？
  关键词：「取消」「中止」「停止执行」「abort」「cancel」
  → 有：✅
  → 没有：⚠️ 标注「确认节点无拒绝/超时处理逻辑」
```

---

### L3：复杂度评分

```
复杂度得分 = (SKILL.md 行数 / 50) + (模块/章节数 × 2) + (硬性规则数 × 0.5)

统计方式：
- SKILL.md 行数：直接统计
- 模块/章节数：统计 ## 二级标题数量
- 硬性规则数：统计含「必须」「禁止」「不得」「must」「must not」「shall not」的行数
```

输出：

| 指标 | 数值 | 评级 |
|------|------|------|
| 总行数 | N | 正常/偏多/过多 |
| 二级标题数 | N | - |
| 硬性规则数 | N | - |
| 复杂度得分 | N | ✅ <10 正常 / ⚠️ 10-20 偏复杂 / ❌ >20 建议重构 |

---

### L4：冗余规则自检

对 SKILL.md 中每条 P1/P2 规则（含「必须」「禁止」「需要」的句子），逐条问：

> 「如果删掉这条规则，在真实对话场景中会出现什么具体问题？」

判断标准：
- 「模型本来就会这样做，无需显式规定」→ ⚠️ 冗余候选
- 「有具体的业务数据或场景特殊性」→ ✅ 保留必要
- 「只是重复了其他规则的意思」→ ⚠️ 冗余候选

输出冗余候选列表（仅供人工复核，不自动删除）。

---

### L5：规则可测试性

对每条硬性规则，检查是否可以被测试用例验证：

```
可测试 → 有具体的输入/输出/工具调用可验证 ✅
模糊难测 → 「尽量」「尽可能」「适当」等主观表述 ⚠️，建议改写为可量化描述
不可测 → 只描述意图而无行为 ❌，标注「无法设计断言」
```

```
- ✅ 通过：[N]  ⚠️ 建议改进：[N]  ❌ 需要修复：[N]

### L1 description 完整性
[逐项结果]

### L2 HiL 节点
[检查结果，无不可逆操作时：「未发现不可逆操作，跳过」]

### L3 复杂度
[表格]

### L4 冗余规则（候选）
[列表，或「未发现明显冗余」]

### L5 可测试性
[列表]

### 改进优先级
P0（必须修复）：[❌ 项目]
P1（建议修复）：[⚠️ 项目]
```

---

## 准则

- 静态分析，不运行任何 subagent 或 MCP 工具
- 所有判断引用 SKILL.md 原文，标注行号
- ⚠️ 是建议，❌ 是必须修复，区别对待
- 不主动修改 SKILL.md，只输出报告
